# 能力检查

## 实体

**函数**是基本实体。每个函数有一个**名字**，名字中编码了该函数的**能力集合**。

## 能力

能力是函数的运行时性质的声明，共有八种：

| 能力 | 含义 | 参与调用规则 | 推断方式 |
|------|------|------------|---------|
| A | 异步执行 | 否（签名推断） | `async fn` |
| B | 可能阻塞线程 | 是 | 传播 |
| I | 执行 I/O 操作 | 是 | 传播 |
| M | 修改共享可变状态 | 否（签名推断） | `&mut` 参数 |
| P | 端口（接口方法） | 是 | trait 名推断 |
| S | 有副作用（读写全局、环境变量、随机等） | 是 | 传播 |
| T | 依赖线程局部状态 | 是 | 传播 |
| U | 包含不安全操作 | 否（签名推断） | `unsafe fn` |

### 函数级别

| 级别 | 能力范围 | 含义 | 测试策略 |
|------|---------|------|---------|
| pure | 空集 | 确定性、无副作用 | 直接单元测试 |
| good | A/B/M 子集 | 无 I/O、无副作用，可单元测试 | 直接单元测试 |
| ok | A/B/M/P 子集 | good 的超集；P 方法可通过 mock 测试 | mockall mock 测试 |

## 命名约定

函数名以 `rvs_` 开头。名字的最后一节（以下划线分隔）若全由能力字母组成，则视为能力后缀。

例：
- `rvs_add` → 无能力
- `rvs_divide` → 无能力
- `rvs_write_db_ABI` → 能力 {A, B, I}

### P（Port）能力

P 是特殊的能力，不从函数签名推断，也不从后缀字母解析。

当当前 crate 中定义的 trait 名字以 `Repository`、`Client`（可扩展列表）结尾时，该 trait 被标记为 **Port**。Port trait 的所有方法**自动获得且仅获得 P 能力**——不参与投票、不传播 I/S/T/B 等。调用 Port 方法的函数会通过传播获得 P。

P 的语义：这是一个接口（端口）方法，调用者通过依赖注入使用它，而非直接依赖具体实现。由于实现可被替换为 mock，因此标记为 P 的函数可以通过 mock 进行测试。

### 调用规则中的 P

P 参与调用规则并向上传播：没有 P 的函数不能调用有 P 的函数。这确保了所有通过 Port 接口进行 I/O 的代码都会被标记为需要依赖注入。

## 调用规则

函数 A 调用函数 B 时，A 的能力集必须是 B 的能力集的超集。

换言之：B 有的每一个能力（除 A/M/U 外），A 都必须有。

**"我有，方可调你。"**

A/M/U 三个签名推断能力不参与调用规则——它们只从函数自身的签名推断，不通过传播获得。P 参与调用规则并向上传播。

## 处理流程

1. 扫描所有 `.rs` 源文件
2. 识别所有以 `rvs_` 开头的函数定义
3. 从函数名中萃取能力集合
4. 在函数体中识别所有对 `rvs_` 函数的调用（包括直接调用和方法调用）
5. 对每对调用关系检查能力合规性
6. 报告所有违规
